# AI 小说转剧本工具需求设计

## 1. 背景与目标

本项目面向七牛云 XEngineer 72 小时作品挑战赛题目“AI 小说转剧本工具”。很多小说作者希望将自己的作品改编成剧本，但从小说到剧本需要完成章节梳理、人物提炼、场景拆分、对白重写、动作描写和结构化输出，人工成本较高。

本工具的目标是提供一款小说转剧本 AI Agent Web 应用：用户输入 3 个章节以上的小说文本后，系统由智能体按 ReAct 架构完成章节解析、故事理解、场景规划、剧本写作、YAML 校验和格式修复，最终生成结构化 YAML 剧本初稿，让作者可以快速获得可编辑、可复制、可下载、可继续打磨的改编结果。

GitHub 仓库地址：<https://github.com/jigexiadechoudan/zen-story2script.git>

## 2. 用户与使用场景

### 2.1 目标用户

- 小说作者：希望快速判断作品是否适合改编为短剧或影视剧本。
- 编剧助理：需要从长文本中整理角色、场景和对白初稿。
- 短剧团队：希望把连载小说片段快速拆成可讨论的分场剧本。
- 内容工作室：需要批量评估故事改编潜力，并沉淀结构化素材。

### 2.2 核心场景

1. 作者将 3 章以上小说文本粘贴到页面。
2. 作者填写作品标题，并选择改编目标，例如短剧、影视剧本或分场大纲。
3. 系统校验章节数量，章节不足时提示补充文本。
4. 系统调用 AI 完成角色抽取、剧情梳理、场景拆分和 YAML 剧本生成。
5. 作者在页面查看 YAML 结果，复制或下载文件。
6. 作者根据 YAML 中的角色表、场景列表、对白和改编说明继续人工打磨。

## 3. MVP 功能需求

### 3.1 小说输入

- 支持粘贴纯文本小说。
- 支持填写作品标题。
- 支持选择改编目标：
  - `short_drama`：短剧。
  - `screenplay`：影视剧本。
  - `scene_outline`：分场大纲。
- 支持填写风格提示，例如“现实主义”“悬疑”“轻喜剧”“古装权谋”。
- 系统自动识别章节标题，兼容以下常见格式：
  - `第1章`、`第一章`、`第 1 章`。
  - `Chapter 1`。
  - 以空行和明显标题分隔的章节片段。
- 系统必须校验至少 3 个章节；不满足时不进入 AI 转换。

### 3.2 AI 剧本转换

- 系统将小说文本转换为结构化 YAML，而不是只输出自然语言总结。
- 转换过程包含以下阶段：
  - 章节摘要：提炼每章主要事件、冲突和情绪变化。
  - 角色抽取：识别主要角色、身份、性格、关系和人物弧光。
  - 场景拆分：将小说事件改写为可拍摄的场景。
  - 剧本生成：输出动作、对白、括号提示、转场等剧本元素。
  - 改编说明：解释主要删改、合并、重排和保留原因。
- AI 输出必须符合项目定义的 YAML Schema。
- 如果 AI 首次输出不是合法 YAML，系统自动尝试修复一次。
- 后端整体遵循 ReAct 架构：智能体先根据当前状态做决策摘要，再调用工具，观察工具结果，然后决定继续分析、生成、校验、修复或结束。
- 页面可展示 Agent 步骤摘要，例如“已完成章节解析”“已抽取角色和关键事件”“YAML 校验通过”，但不展示模型隐藏推理链。

### 3.3 YAML 结果预览与导出

- 页面展示生成后的 YAML 文本。
- 支持一键复制 YAML。
- 支持下载 `.yaml` 文件。
- 展示基础质量说明：
  - 识别到的章节数。
  - 生成的角色数。
  - 生成的场景数。
  - 可能需要人工打磨的提示。

### 3.4 Schema 文档入口

- README 和页面中提供 YAML Schema 文档入口。
- 用户可以理解每个字段的含义和设计原因。
- Schema 文档应说明哪些字段是必填、哪些字段用于后续扩展。

### 3.5 示例与演示

- 项目提供示例小说输入和示例 YAML 输出。
- 无 AI API Key 时，README 仍能通过示例文件展示预期效果。
- Demo 视频需要覆盖：
  - 输入 3 章以上小说。
  - 触发转换。
  - 查看 YAML。
  - 下载或复制结果。
  - 简要说明 YAML Schema 的设计价值。

## 4. 非功能需求

### 4.1 可运行性

- 评委应能根据 README 在本地启动前后端。
- 后端使用 JDK 21 + Spring Boot + Spring AI。
- 后端采用单个 Spring Boot 应用工程，不拆 Maven 多模块；内部按 `api`、`agent`、`tools`、`schema`、`config` Java 子包组织。
- AI 接入通过 Spring AI ChatClient 统一封装，默认支持 OpenAI 兼容接口，通过环境变量配置：
  - `OPENAI_BASE_URL`
  - `OPENAI_API_KEY`
  - `OPENAI_MODEL`
- 前端使用 Vite + Vue 3。
- MVP 首版采用无数据库设计，转换结果由接口直接返回；后续可扩展云端持久化、历史版本和作品管理。

### 4.2 稳定性

- 章节不足时返回明确错误。
- AI Key 缺失时返回配置错误，并提示使用示例输出查看效果。
- AI 调用超时或失败时返回可读错误。
- YAML 解析失败时自动修复一次；修复失败时返回错误信息和原始片段，便于排查。

### 4.3 可编辑性

- 输出使用 YAML，是因为 YAML 对非技术用户更易读、易复制、易局部修改。
- 有顺序的内容必须使用数组，例如场景列表、剧情节拍、对白块。
- 输出不应混入 Markdown 代码块标记，避免下载后无法直接解析。

### 4.4 MVP 首版边界与后续扩展

- 账号系统首版暂缓，后续可扩展作者登录、作品草稿箱和个人配置。
- 用户输入首版仅用于当次转换，后续可扩展历史记录、版本对比和云端保存。
- 多人协作首版暂缓，后续可扩展评论、批注、协同编辑和角色分工。
- 版权与敏感内容检测首版仅在 README 中提示用户确认文本授权，后续可扩展授权声明、风险提示和内容检查。
- AI 输出首版定位为可编辑初稿，后续可扩展人工审校流程、专业排版和生产级导出。
- MVP 后端按小说转剧本 AI Agent 项目组织：由 `NovelToScreenplayAgent` 按 ReAct 循环调用章节解析、故事分析、场景规划、YAML 写作、Schema 校验和 YAML 修复工具，而不是一次性调用模型直接改写全文。

## 6. 赛事合规与 PR 规范

### 6.1 知识产权与发布边界

- 作品知识产权归提交队伍或个人所有。
- 用户输入的小说文本应由用户自行确认授权，README 需提示不得输入未获许可的第三方作品用于公开发布或对外分享。
- 示例小说应使用原创短文本，避免引用受版权保护的小说片段。
- Demo 视频、README 截图和示例输出只展示原创示例文本与生成结果。
- 如后续扩展云端存储、公开分享或团队协作，需要增加授权确认、内容风险提示和删除机制。

### 6.2 学术诚信与原创说明

- 项目应保持自主完成，不引入抄袭代码或未说明来源的历史代码。
- 如复用自己过去的代码片段，必须在对应 PR 描述中注明来源和改动范围。
- 引用第三方库、框架、模板或生成工具时，必须在 README 中列明依赖、用途和项目原创功能部分。
- 每个 PR 描述应说明该 PR 的原创实现内容，避免出现描述空白或与实际代码变更严重不符。
- 开发过程中避免大段复制外部实现，降低代码重复率风险。

### 6.3 PR 提交规范

每个 PR 只做一件事，标题和描述必须清晰完整。推荐模板：

```text
标题：一句话说明本 PR 新增/修改了什么

功能描述：
- 说明该功能的作用与使用方式。

实现思路：
- 简要说明技术选型或核心实现逻辑。
- 如引用第三方库、模板或复用历史代码，在这里注明来源。

测试方式：
- 说明如何验证功能正常运行。
- 写明执行过的命令、接口请求或手动验收步骤。

原创与依赖说明：
- 本 PR 原创实现的部分。
- 新增第三方依赖及其用途。
```

### 6.4 持续交付要求

- 从议题发布之日起保持持续 PR 和 commit 提交。
- 大功能拆成多个独立 PR 分步提交，避免临尾突击提交。
- 主分支在每个 PR 合并后保持可运行，评委任意时间查看都能复现核心演示效果。
- 所有 commit 时间戳必须落在所选批次的开始与截止时间内。
- README、文档、示例和 demo 视频链接也应通过 PR 持续补充。

## 7. 验收标准

### 7.1 功能验收

- 输入 3 章以上小说文本后，可以生成结构化 YAML 剧本。
- 输入不足 3 章时，页面和接口都能提示错误。
- 输出 YAML 至少包含：
  - 作品信息。
  - 改编目标。
  - 角色表。
  - 剧情大纲。
  - 场景列表。
  - 动作、对白、括号提示和转场。
  - 改编说明。
- 用户可以复制和下载 YAML。
- README 可以指导评委完成一次完整演示。

### 7.2 文档验收

- `docs/requirements.md` 说明需求、范围、验收标准和比赛交付策略。
- `docs/architecture.md` 说明 Java 后端架构、API、AI 流程和 YAML Schema。
- `docs/mvp-workplan.md` 说明多线推进模块、任务边界和 PR 顺序。
- README 引用三份文档，并列明第三方库或框架、依赖用途和原创功能部分。

### 7.3 演示验收

- Demo 视频有声音讲解。
- 视频展示主要功能和效果。
- 视频中说明 AI 如何把小说拆成角色、场景和剧本 YAML。
- 视频中说明 Agent 的 ReAct 工作方式：先分析任务状态，再调用工具，观察结果，最终生成并校验 YAML。
- README 中的视频链接可访问、可播放。

## 8. PR 拆分建议

为了满足比赛“全周期持续交付”的要求，建议按以下粒度拆分 PR：

1. `docs: add requirements and architecture`
   - 新增需求文档、架构文档和 YAML Schema 说明。
2. `chore: initialize backend and frontend`
   - 初始化 Spring Boot 后端和 Vue 前端项目。
3. `feat(backend): parse and validate novel chapters`
   - 实现章节切分和不少于 3 章校验。
4. `feat(backend): add OpenAI compatible client`
   - 实现 AI 接口配置和调用。
5. `feat(backend): generate and validate screenplay yaml`
   - 实现 Prompt、YAML 校验和修复流程。
6. `feat(frontend): add conversion workspace`
   - 实现输入、提交、状态展示和结果预览。
7. `feat(frontend): add yaml copy and download`
   - 实现复制、下载和质量说明。
8. `docs: add README demo guide`
   - 补充依赖说明、启动方式、示例和 demo 视频链接。

## 9. 后续扩展规划

以下能力不纳入 72 小时 MVP 的首版交付，但可以作为后续迭代方向：

- 用户注册和登录：支持作者保存个人作品、管理草稿和转换记录。
- 云端持久化存储：保存小说输入、剧本输出、历史版本和导出文件。
- 多人协作编辑：支持作者、编剧、导演在同一剧本上评论和修改。
- 版权与敏感内容辅助检查：提示用户确认文本授权，并对潜在风险内容给出提醒。
- 专业排版格式导出：支持 PDF、Final Draft、Fountain、Word 等格式。
- 长篇小说全集处理：支持更长文本的分批解析、分卷改编和跨章节一致性检查。
- 可视化剧本编辑器：将 YAML 转换为角色表、分场表和对白编辑界面。
